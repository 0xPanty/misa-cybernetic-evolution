# v0.26 Cybernetic Architecture Roadmap

## 文档定位

这不是新增功能清单，是**架构盘点 + 控制论意义上的空洞修补图**。

v0.2 到 v0.25 这一线已经把"侧车学习层"的**骨架**搭出来了：5 路 route 表、L0-L4 验证闸、control contract 模板、damping 路、tournament gate、observe-only Hermes 适配器。继续在这条线上加 v0.27、v0.28、v0.29 的可改进余地不大——会变成给一个已经完整的骨架不停糊新皮肤。

v0.26 这一线要做的是把**钱学森工程控制论的六个核心元件**对照现状一对一审视：
- 哪些已经有，叫什么名字；
- 哪些只有"形状"没有"实体"；
- 哪些是经验值代替了控制论判据；
- 哪些根本没定义。

补完这六个空洞，这个项目才从"漂亮的清单"升级成"真正的受控自进化系统"。

## 控制论六要素 vs 现状盘点

| 控制论要素 | 项目里对应的位置 | 现状 |
| --- | --- | --- |
| 设定值 setpoint | `control_contract.primary_setpoint`（自由文本） | 有概念，**不可测** |
| 观测 measurement | observation plane + `examples/misa-learning/*.fixture.json` 里的 `signals[]` | 有 schema，**传感器缺位** |
| 偏差驱动 error → action | `scripts/lib/learning-loop.mjs::classifyRoute()` 的 5 路分类 | 已实现，确定性，但**判据是 signal 关键词匹配** |
| 受限执行器 bounded actuator | `control_contract.actuator_budget` + `scripts/lib/governance.mjs::HIGH_RISK_PATTERNS` | 有形状，**靠正则识别，强度不足** |
| 阻尼 damping | 5 路里的 `damping` 路 + `damping_rules.json` | 已实现，**阈值是经验值** |
| 先稳定后优化 | 默认 zero-call / draft only / observe-only | 全仓贯彻得很好 |
| 分层控制 | L0-L4 verification gates | **是串行验证，不是多时间尺度控制** |

**结论**：骨架对得上六要素，但其中四件（setpoint、observation、actuator、damping）只有形状没有控制论强度，加上**闭环反馈缺失**和**plant model 未定义**两个隐性空洞，构成本路线图要补的六块。

## 空洞 1：setpoint 不是可测量量

### 现状
`schemas/control_contract.schema.json` 的 `primary_setpoint` 是 `string`，由人填自由文本。`docs/CONTROL_CONTRACT.md` 给的好例子是"reduce repeated tool-selection failures for mail triage"——好的方向描述，**但控制器没法回答"现在离设定值多远"**。

### 病灶
控制论的 setpoint 必须是**可测标量或向量**，因为闭环反馈靠的是"测量值 − 设定值 = 误差"。一段自由文本算不出误差，整个反馈链是开环。

### 补法
- 改 `schemas/control_contract.schema.json`：`primary_setpoint` 字段升级为对象：
  ```json
  {
    "metric_id": "string (必须在 metric registry 里)",
    "target_value": "number",
    "tolerance": "number",
    "direction": "minimize | maximize | hold_within"
  }
  ```
- 新增 `schemas/metric-registry.schema.json` 和 `docs/metric-registry.md`：定义所有可控指标的命名、单位、采样方法、采样窗口。例：
  - `mail_triage.tool_selection_failure_rate`（采样 window=24h，单位=比例）
  - `farcaster.private_memory_leak_count`（采样 window=session，单位=count）
  - `provider.timeout_rate`（采样 window=1h，单位=比例）
- `governance.evaluateControlContract()` 增加一条 violation：`primary_setpoint.metric_id` 未在 registry 里时拒绝。
- 旧的自由文本 setpoint 保留为 `setpoint_narrative`（人话描述），但**不再是控制器判据**。

### 落地动作
- 新 schema + registry 的最小版（先注册 3-5 个核心 metric 就行）
- `examples/control_contract.example.json` 改用新结构
- `governance.test.mjs` 加用例：自由文本 setpoint → violation

### 已落地
- `schemas/metric_registry.schema.json` + `examples/metric_registry.example.json` 已建立最小 metric registry。
- `schemas/control_contract.schema.json` 已把 `primary_setpoint` 升级为 `{ metric_id, target_value, tolerance, direction }`，人话目标移到 `setpoint_narrative`。
- `governance.evaluateControlContract()` 已拒绝自由文本 setpoint、未注册 metric、方向不匹配的 setpoint，以及未注册 guardrail metric。
- `signal_extractor.recall` / `signal_extractor.precision` 已进入 registry，后续 metric registry 扩展时可直接挂 P2 的 fixture review 结果。

## 空洞 2：传感器（signals extractor）缺位

### 现状
`learning-loop.mjs::classifyRoute()` 完全依赖 event 里**预打好的** `signals: ["reusable_workflow", "public_posting_boundary", ...]`。这些 signals 现在**只有 fixture 里手写**（`examples/misa-learning/*.fixture.json`）。从真实 Misa session 到 signals 的传感器**整个不存在**。

### 病灶
控制论意义上：**系统不可观（unobservable）**。控制器（路由表）再确定性、再漂亮，传感器没接，整个回路是开环。这是目前**最大的工程空洞**。

### 补法
分两层：

**第一层：确定性传感器（必须做）**
- 新增 `scripts/lib/signal-extractor.mjs`：输入一段 session 文本 + 元数据，输出 `{ signals: [...], evidence_count, confidence }`，可直接喂给 `simulateLearningCycle()`。
- 规则数据化：`scripts/lib/signal-rules.json` 定义每个 signal 的触发条件（关键词集合、上下文要求、negative 条件），引擎跑表。加 signal = 改 JSON，不改 .mjs。
- 第一批至少覆盖现有 6 个 route 路径用到的 signal：`public_posting_boundary`、`farcaster_public_memory_risk`、`reusable_workflow`、`repeated_failure_pattern`、`single_failure`/`avoid_overreaction`、`stable_user_preference`/`stable_project_fact`、`candidate_replay_failed`。
- **不调用 LLM**。同一输入必须永远产出同一组 signals。

**第二层：LLM 候选标注（可选，未来做）**
- LLM 可以**提名**候选 signal，但必须经过第一层的确定性归一化才能进 route 表。
- 实现成一个独立 stage：`llm_signal_proposal -> deterministic_filter -> signals[]`。LLM 只能新增候选，不能改判据。

### 落地动作
- 这件事我前面在 P2 工单已经写过（最小 signal extractor），这里把它**升级为架构层一等公民**：每个 fixture 必须同时保留 `signals_hand`（手写真值）和 `signals_extracted`（机器抽取结果），CI 检查 `signals_extracted ⊇ signals_hand`，差距收敛到 0 才算传感器达标。
- 新增 metric：`signal_extractor.recall`、`signal_extractor.precision`，进 metric registry。

## 空洞 3：闭环验证缺失（最关键的一块）

### 现状
进化链：`evidence → route → candidate → tournament → winner → 人审 → 发布`。但**发布之后没有反向测量**——一个 skill 升级了，下周这个 skill 真的让对应 setpoint 推近了吗？没人回答。

`evolution-tournament-gate.mjs` 里的 `experience_ledger` 只是"哪些候选赢了、哪些输了"的横向比较，**不是发布后的纵向回测**。

### 病灶
这是**真自进化和自证预言的分水岭**。没有 post-deploy measurement，所谓"正向改动"只是候选打分系统说它好——候选打分系统本身没人审，就是循环论证。

### 补法
- 每个被发布的 candidate（不论 skill / memory / case / policy）**必须挂一个 deployment ticket**：
  ```json
  {
    "deployment_id": "...",
    "candidate_id": "...",
    "setpoint_metric_id": "...",          // 引用空洞 1 的 metric
    "pre_deploy_value": 0.18,
    "target_value": 0.05,
    "measurement_window": "7 days / 50 sessions",
    "decision_at_window_end": "pending"
  }
  ```
- 新增 `scripts/lib/post-deploy-measurement.mjs`：到期自动回测，结果分类为：
  - `confirmed_positive`：metric 推近 setpoint，候选保留
  - `confirmed_negative`：metric 远离 setpoint，自动 rollback + 进 damping
  - `null_effect`：metric 没动，候选保留但记一笔"低 ROI"
- 这是**自进化的真闭环**。没有这一环，前面 L0-L4 全是开环漂亮话。

### 落地动作
- 新 schema `schemas/deployment-ticket.schema.json`
- 新 route：`post_deploy_review`（不是新的进化方向，是回测结果的处理路径）
- `evolution-tournament-gate.mjs` 增加一个 input：`historical_post_deploy_results`，让 tournament 知道哪类候选历史回测好/差

### 落地状态
- 已落地：deployment ticket schema + example。
- 已落地：`scripts/lib/post-deploy-measurement.mjs`，纯规则测量，三分结果为 `confirmed_positive` / `confirmed_negative` / `null_effect`。
- 已落地：负向结果只推荐 rollback + damping，不执行生产回滚，`production_authority=false`、`llm_api_calls=0`。
- 已落地：`evolution-tournament-gate.mjs` 接收历史 post-deploy 结果，并作为 advisory pressure 暴露，不改变 route owner / winner / production boundary。

## 空洞 4：actuator_budget 强度不足

### 现状
`governance.mjs::HIGH_RISK_PATTERNS` 用正则识别高风险动作（`/public\s+post/i`、`/timer/i`、`/delete/i` 等）。actuator_budget 是自由 string 数组。

### 病灶
正则识别只挡得住"诚实的违规"。把 actuator 写成 "send to channel"、"schedule background"、"remove records"——全部识别不到。强度等于"团队自觉用对词"。

### 补法
（这部分已经在 P3 工单写过，这里把它正式纳入架构层）

- 新 schema `schemas/actuator-enum.json`，定义全部合法 actuator + 高风险子集
- `control_contract.actuator_budget` 改为 enum 数组，未知 actuator 直接 violation
- `HIGH_RISK_PATTERNS` 删除，改成 `Set` 比对

### 落地动作
照原 P3 工单执行。这一块技术难度最低，但**架构强度提升最直接**——actuator 从模糊文字升级成显式契约。

## 空洞 5：没有显式稳定性判据

### 现状
`damping_rules.json` 用经验阈值（evidence_count≥2、3 successes 才升级、2 failures 才进 case）。控制论里你不能光说"加阻尼"，要证明系统**不会发散**。

### 病灶
"经验阈值能用"和"稳定性可证明"不是一回事。当系统规模放大、route 数变多、外部输入分布漂移时，经验阈值会失效，且**没人知道什么时候失效**。

### 补法
- 给每个 route 定义一个**发散预警指标**（divergence indicator）。例：
  - `memory_route.promote_rollback_ratio`（过去 N 周期 memory 升级被回滚的比例）
  - `skill_route.replay_failure_streak`（连续 N 个 skill 候选 replay 失败）
  - `policy_route.approval_rejection_rate`（policy 候选被人审拒绝的比例）
- 新增 `scripts/lib/stability-monitor.mjs`：周期性计算各 route 的 divergence indicator，超过预警阈值 → 整个系统进入 **safe mode**：
  - 所有 route 冻结，只接受 `damping` 和 `ignore`
  - 触发原因记入 `runs/stability-incidents/`
  - 必须人工解除才能退出 safe mode
- 这是**控制论意义上的稳定性闸**。它不能阻止系统出错，但能阻止系统在错误轨道上继续加速。

### 落地动作
- 新 schema `schemas/stability-indicator.schema.json`
- 一个最小版 monitor，先盯 memory 和 skill 两条最容易发散的路
- safe mode 状态机进 README "Current Boundary" 表

### 落地状态
- 已落地：`schemas/stability-indicator.schema.json` + example。
- 已落地：`scripts/lib/stability-monitor.mjs`，先盯 `memory_route.promote_rollback_ratio` 和 `skill_route.replay_failure_streak`。
- 已落地：超过 safe-mode 阈值时输出 `allowed_routes=["damping","ignore"]`，冻结 `memory/skill/case/policy`，并要求人工解除。
- 已落地：CLI 可显式写入 `runs/stability-incidents/`；默认不碰 production/VPS，不改 live route table，`llm_api_calls=0`。

## 空洞 6：分层控制没分时间尺度

### 现状
L0-L4 是**串行验证闸**（static → replay → shadow → canary → publication），每个候选依次通过。这是 staged release，不是控制论意义上的"hierarchical control"。

### 病灶
经典控制论：**内环快、外环慢，各管各的稳定性**。一个真正的自进化系统至少要有三个时间尺度，否则慢变量（setpoint 漂移、用户意图变化）和快变量（单次 actuator 校验）混在一起，慢的会把快的拖垮，快的会让慢的失真。

### 补法
明确分三层，每层有自己的 setpoint、自己的稳定性指标、自己的更新频率：

| 层级 | 时间尺度 | 管什么 | 现状对应 |
| --- | --- | --- | --- |
| 快环 inner | 毫秒–秒 | 单次 actuator 是否在 budget 里、单次输出是否过 redaction 闸 | 已有（governance.mjs 实时检查） |
| 中环 middle | 小时–天 | 候选评估、tournament、人审、post-deploy measurement | 部分有（evolution-tournament-gate + 待补 post-deploy） |
| 慢环 outer | 周–月 | setpoint 自身是否还合理、route 表本身的判据是否还有效、metric registry 是否需要新增 | **完全缺失** |

**慢环可以推翻快环**：例如"最近一周快环一直在通过 mail_triage 的某类 actuator，但 `mail_triage.tool_selection_failure_rate` 这个 setpoint 完全没改善"——这是快环判据本身需要重校的信号，慢环要能下发这个判据更新。

### 补法（具体）
- 新 doc `docs/control-hierarchy.md`，把现有命令按三层分类（doctor / smoke / precheck 是哪一层，calibration / health 是哪一层，evolution-tournament 是哪一层）
- 新增慢环模块 `scripts/lib/outer-loop-review.mjs`：周期性回看（建议 7 天为周期）setpoint vs 实际 metric 走势，输出"setpoint 是否需要调整 / route 判据是否需要重校 / metric registry 是否需要扩"的建议
- 慢环输出**只生成建议、不自动改判据**——判据修改必须人审，因为这是 one-way door

### 落地动作
- 先写 `docs/control-hierarchy.md`，把现有功能正确归位
- 再实现 outer-loop-review 的最小版（先只看 metric 走势 + 输出报告，不做任何自动操作）

## 隐藏空洞 0：plant model 未定义

这条放最后，因为它是所有空洞的**前置**。

### 现状
项目里反复说"侧车控制 Misa/Hermes 这个 agent"。但**被控对象到底是谁、状态变量是哪些、输入是什么、扰动来自哪里**——没有正式定义。

### 病灶
钱学森讲：控制前要先**辨识被控对象**。没有 plant model，控制器是在控一个虚空的影子。所有 setpoint、metric、actuator、route 都缺一个统一坐标系。

### 补法
在 `ARCHITECTURE.md` 顶部加一节 **"Plant Definition"**，明确：

```
Plant = Misa runtime agent + its conversation environment

State variables (state vector x):
- memory consistency rate
- skill hit rate by channel
- public-channel safety incident count
- provider health (per provider: timeout rate, error rate, latency)
- user satisfaction proxies (correction rate, takeover rate)

Inputs (control vector u):
- new memory / skill / case / policy artifacts (from this sidecar)
- actuator-budgeted writes

Disturbances (w):
- LLM stochasticity (gacha)
- provider drift
- user intent drift
- external framework updates (Hermes/Zilliz schema changes)

Output (y):
- agent visible behavior
- user feedback signal
- redacted session evidence
```

这一段写清楚，前面五个空洞才有共同的坐标系。**没有 plant model，metric registry 和 setpoint 都漂浮着**。

### 落地动作
- ARCHITECTURE.md 顶部加 Plant Definition 节
- 每个 metric 注册时必须声明它测的是 plant state 的哪个分量
- 每个 actuator 必须声明它作用在 plant 的哪个输入维度

### 已落地
- `ARCHITECTURE.md` 已新增 Plant Definition，明确 plant、state vector、control inputs、disturbances、outputs 和禁止越权面。
- `schemas/plant_model.schema.json` + `examples/plant_model.example.json` 已把 plant model 变成机器可校验契约。
- `scripts/lib/plant-model.mjs` 会检查所有 actuator enum 是否都映射到 plant input dimension。
- `scripts/lib/metric-registry.mjs` 会检查每个 metric 是否指向 plant state component。

## 推荐执行顺序

按依赖关系排，**不要并行**：

```
空洞 0 (Plant Definition)
        ↓ 所有后续的坐标系
空洞 1 (setpoint 可测化) + 空洞 4 (actuator 枚举化)
        ↓ 控制器有了量纲
空洞 2 (信号传感器)
        ↓ 闭环的输入端打通
空洞 3 (post-deploy 闭环)
        ↓ 闭环的输出端打通
空洞 5 (稳定性判据)
        ↓ 闭环不会发散
空洞 6 (分层时间尺度)
```

**起码先做 0 → 1 → 4 → 2 这四步**，前四步走完，这个项目就从"漂亮的开环建议系统"变成"最简但闭合的受控学习系统"。空洞 3、5、6 是稳定运行后的深化。

每一步建议独立分支独立 PR：`feat/plant-definition`、`feat/measurable-setpoint`、`feat/actuator-enum`、`feat/signal-sensor`……不要叠在一起。

## 不做什么

- **不再加 v0.27、v0.28、v0.29 这种"再糊一层皮肤"的版本**。v0.26 这一线必须收敛到上述六个空洞，做完才能开 v0.27 主题。
- **不引入新的 carrier 适配器**（已有 Hermes、LangGraph、OmniAgent 就够了，加新 carrier 不增加控制论强度）。
- **不让 LLM 进入任何 route 决策、metric 测量、stability 判定环节**。LLM 只能在"候选生成"和"人话翻译"两个位置，永远在控制器之外。
- **不在没有 plant model 的情况下急着加新 metric**，否则又是一堆漂浮的数字。

## 一句话总结

v0.2 到 v0.25 把"侧车学习层"的**骨架**搭完了。v0.26 这一线要做的是**把骨架从"控制论形状"升级成"控制论实质"**：让 setpoint 可测、传感器接上、闭环跑通、actuator 强类型、稳定性可证、分层有时间尺度、plant 有定义。

补完这六块，这个项目才能正式回答你最初那个问题：**"凭什么相信这次进化是正向的，而不是又一次野蛮生长？"**——因为我们能测、能算误差、能闭环验证、能在发散前刹车。这是钱学森式工程控制论给"自进化 agent"行业的真正贡献，不是花哨的命名，而是**让一个抽卡式的系统在工程意义上变得可证明地稳定**。
